# 1. Why Mamba, and why now?

Sequence modeling has a practical tension:

1. We want a model to reason over long contexts.
2. We need training and inference to be efficient enough to run on real hardware budgets.

Transformers solve (1) very well, but attention has quadratic cost in sequence length. If you double context length, attention work grows by roughly 4x. That quickly becomes painful for latency and memory.

Mamba's central idea is to recover **linear-time sequence processing** while preserving strong content-dependent behavior. It does that by revisiting state space models (SSMs), then adding a selective mechanism that conditions state updates on the current token [1].

<div class="box">
  <div class="box-title">Key idea in one line</div>
  Mamba keeps a recurrent hidden state like an SSM, but makes the update rule input-dependent so the model can decide what to retain or forget at each step.
</div>

<div style="text-align: center; margin: 1.5rem 0;">
  <img src="/assets/img/posts/mamba4080_tutorial/attention_vs_ssm_scaling.png" alt="Scaling comparison between attention and recurrent SSMs" style="max-width: 82%;">
</div>


<details markdown="1">
<summary>Bridge: Why not just use an RNN then?</summary>

Classic RNNs are linear-time too, but they often struggle with long-range dependencies and stable optimization. Modern SSM parameterizations (S4 lineage) give a more structured, better-conditioned recurrent system. Mamba builds on that lineage and adds selectivity for stronger content-dependent behavior [1,2].

</details>

# 2. Notation and shape conventions

We will use these symbols throughout.

- \(B\): batch size
- \(L\): sequence length
- \(d_{inner}\): inner channel dimension (SSM channels)
- \(N\): state dimension per channel
- \(d_{model}\): model embedding width

Main tensors:

- input sequence: \(u \in \mathbb{R}^{B \times L \times d_{inner}}\)
- per-token step sizes: \(\Delta \in \mathbb{R}^{B \times L \times d_{inner}}\)
- continuous dynamics: \(A \in \mathbb{R}^{d_{inner} \times N}\)
- selective input map: \(B_t \in \mathbb{R}^{B \times d_{inner} \times N}\)
- selective readout map: \(C_t \in \mathbb{R}^{B \times d_{inner} \times N}\)
- recurrent state: \(x_t \in \mathbb{R}^{B \times d_{inner} \times N}\)
- output: \(y \in \mathbb{R}^{B \times L \times d_{inner}}\)

We will often highlight critical objects with color:

$$
\textcolor{#4ecdc4}{A},\;\textcolor{#ff9f6e}{\Delta_t},\;\textcolor{#7ee787}{x_t}
$$

# 3. State space models from first principles

## 3.1 Continuous-time view

A linear state space model is typically written as

$$
\dot{x}(t) = \textcolor{#4ecdc4}{A}x(t) + B_c u(t),
\qquad
y(t) = C_c x(t) + d_{\mathrm{skip}} \odot u(t).
$$

Intuition:

- \(x(t)\) is internal memory.
- \(A\) controls how memory decays or oscillates.
- \(B_c\) injects current input into memory.
- \(C_c\) reads memory into output.
- \(d_{\mathrm{skip}}\) is a direct skip path from input to output.

For sequence models, we need a discrete-time update per token.

## 3.2 Discrete-time recurrence used by Mamba-style scan

At token \(t\), with input-dependent step \(\Delta_t\):

$$
\bar{A}_t[b,d,n] = \exp\!\left(\Delta_t[b,d]\;A[d,n]\right),
$$
$$
x_t[b,d,n] = \bar{A}_t[b,d,n]\;x_{t-1}[b,d,n] + \Delta_t[b,d]\;B_t[b,d,n]\;u_t[b,d],
$$
$$
y_t[b,d] = \sum_{n=1}^{N} C_t[b,d,n]\;x_t[b,d,n] + d_{\mathrm{skip}}[d]\;u_t[b,d].
$$

Everything above is channelwise and statewise. In tensor form, the implementation treats these as broadcasted elementwise operations over \((B, d_{inner}, N)\).

<div class="box corollary">
  <div class="box-title">Corollary: Causality is built in</div>
  Because \(x_t\) depends only on \(x_{t-1}\) and \(u_t\), and \(y_t\) depends on \(x_t\), no output at time \(t\) can depend on future tokens. This is why selective scan is naturally causal.
</div>

<details markdown="1">
<summary>Bridge: Why use \(A = -\exp(A_{log})\)?</summary>

In the code, \(A\) is parameterized as negative exponential, which biases dynamics toward stable decay modes. You can see this in `MambaBlock`:

```python
A = -torch.exp(self.A_log)
```

See: [`mamba4080/modules/mamba_block.py`](mamba4080/modules/mamba_block.py)

</details>

# 4. From S4 to Mamba

S4 made state space models practical for long sequence modeling by introducing structured parameterizations and efficient computations [2]. Mamba keeps that high-level idea (an SSM backbone), but introduces a key change:

- S4-style dynamics are mostly fixed after training.
- Mamba makes important terms input-dependent at each token.

In Mamba, \(\Delta_t, B_t, C_t\) are generated from the current sequence features. This lets the model perform **content-dependent state updates** without attention [1].

<div class="box">
  <div class="box-title">S4 vs Mamba (conceptual)</div>
  S4: powerful linear dynamical memory with mostly fixed recurrence.<br>
  Mamba: keeps linear recurrent backbone, but adds selective gating in the recurrence itself.
</div>

<details markdown="1">
<summary>Optional deep dive: Why this matters for language</summary>

Language requires both retention and rapid forgetting depending on context. If recurrence is fixed, the model can struggle to adapt memory policy token-by-token. Selectivity gives a dynamic memory policy that behaves more like content-based routing.

</details>

# 5. The selective scan equations (core of Mamba)

Given

- \(u_t \in \mathbb{R}^{B \times d_{inner}}\)
- \(\Delta_t \in \mathbb{R}^{B \times d_{inner}}\)
- \(B_t, C_t \in \mathbb{R}^{B \times d_{inner} \times N}\)
- \(A \in \mathbb{R}^{d_{inner} \times N}\)
- \(d_{\mathrm{skip}} \in \mathbb{R}^{d_{inner}}\)

we compute:

$$
\bar{A}_t = \exp\big(\textcolor{#ff9f6e}{\Delta_t} \odot \textcolor{#4ecdc4}{A}\big) \in \mathbb{R}^{B\times d_{inner}\times N},
$$

$$
\textcolor{#7ee787}{x_t} = \bar{A}_t \odot x_{t-1} + (\Delta_t \odot B_t) \odot u_t,
$$

$$
y_t[b,d] = \sum_{n=1}^{N} C_t[b,d,n] \, x_t[b,d,n] + d_{\mathrm{skip}}[d] \, u_t[b,d].
$$

In code, the vectorized recurrence is direct:

```python
delta_t = delta[:, t, :].unsqueeze(-1)   # (B, d_inner, 1)
u_t = u[:, t, :].unsqueeze(-1)           # (B, d_inner, 1)
A_bar_t = torch.exp(delta_t * A_expanded)
B_bar_t = delta_t * B[:, t, :, :]
state = A_bar_t * state + B_bar_t * u_t
y_t = torch.sum(C[:, t, :, :] * state, dim=-1)
```

Code: [`mamba4080/ops/selective_scan.py`](mamba4080/ops/selective_scan.py)

<details markdown="1">
<summary>Implementation trick: Why loop over sequence length even in vectorized code?</summary>

The recurrence has true temporal dependence: \(x_t\) depends on \(x_{t-1}\). Full elimination of that loop is nontrivial. The "vectorized" path removes inner Python loops over batch/channel/state but keeps one loop over time.

</details>

# 6. Mamba block end-to-end: intuition, equations, implementation

A single block in this repository follows the Mamba-1 structure.

## 6.1 Block flow

1. Pre-normalize with RMSNorm.
2. Project from \(d_{model}\) to \(2d_{inner}\), split into \(x\)-stream and \(z\)-stream.
3. Run depthwise causal convolution on \(x\)-stream.
4. Generate selective parameters \((\Delta, B, C)\) from convolved features.
5. Run selective scan.
6. Gate scan output with \(\text{SiLU}(z)\).
7. Project back to \(d_{model}\), add residual.

## 6.2 Equations with shapes

Let \(h \in \mathbb{R}^{B \times L \times d_{model}}\).

$$
\tilde{h} = \mathrm{RMSNorm}(h)
$$
$$
[x, z] = W_{in}\tilde{h}, \quad x,z \in \mathbb{R}^{B\times L\times d_{inner}}
$$
$$
x' = \mathrm{SiLU}(\mathrm{DWConv}_{causal}(x))
$$
$$
(\Delta, B, C) = f_{param}(x')
$$
$$
y = \mathrm{SelectiveScan}(u=x', \Delta, A, B, C, d_{\mathrm{skip}})
$$
$$
\hat{y} = y \odot \mathrm{SiLU}(z)
$$
$$
\text{out} = W_{out}\hat{y} + h
$$

## 6.3 Code snippets

```python
xz = self.in_proj(x_norm)  # (B, L, 2 * d_inner)
x_stream, z_stream = torch.split(
    xz, [self.d_inner, self.d_inner], dim=-1
)

x_conv = self.conv(x_stream)  # (B, L, d_inner)
x_conv = F.silu(x_conv)
delta, B, C = self._project_selective_params(x_conv)
A = -torch.exp(self.A_log)  # (d_inner, d_state)

y = selective_scan(u=x_conv, delta=delta, A=A, B=B, C=C, D=self.D,
                   implementation=self.ssm_impl)
y = y * F.silu(z_stream)
out = self.out_proj(y) + residual
```

# 7. Causality and streaming inference

Mamba in this repo supports both:

- full-sequence forward for training
- one-token `step()` for streaming generation

For streaming, each layer carries two states:

1. SSM state: \((B, d_{inner}, N)\)
2. Conv cache: \((B, d_{inner}, K-1)\) for kernel size \(K\)

The single-token update in `MambaBlock.step` follows the same math as full forward. That is tested explicitly.

Code links:

- Block step: [`mamba4080/modules/mamba_block.py`](mamba4080/modules/mamba_block.py)
- Model step: [`mamba4080/models/mamba_lm.py`](mamba4080/models/mamba_lm.py)
- Test: [`mamba4080/tests/test_mamba_block.py`](mamba4080/tests/test_mamba_block.py)

```python
y_full = block(x)
# ... run token-by-token with block.step(...)
torch.testing.assert_close(y_step, y_full, atol=1e-5, rtol=1e-5)
```

<div class="box corollary">
  <div class="box-title">Corollary: generation memory is constant in sequence length</div>
  In step mode, you keep fixed-size recurrent states per layer, not full attention KV caches. This is a major practical advantage for long generation.
</div>

# 8. Selective scan implementations in this project

The repository intentionally keeps three paths:

1. `naive` — easiest to inspect, strongest reference semantics.
2. `vectorized` — practical PyTorch baseline.
3. `triton` — optimized CUDA path with fallback.

All three are dispatched through one entrypoint.

```python
def selective_scan(..., implementation: Literal["naive", "vectorized", "triton"] = "vectorized"):
    if implementation == "naive":
        return selective_scan_naive(...)
    if implementation == "vectorized":
        return selective_scan_vectorized(...)
    if implementation == "triton":
        # try triton, fallback safely when unavailable/unsupported
```

# 9. Triton selective scan: broad view to line-by-line detail

## 9.1 Broad view

The Triton path accelerates the selective scan forward pass on CUDA and uses a custom autograd backward recurrence. It is wrapped in `torch.autograd.Function` so gradients remain explicit and testable.

Code: [`mamba4080/ops/selective_scan_triton.py`](mamba4080/ops/selective_scan_triton.py)

## 9.2 Forward kernel anatomy

Key ideas in the forward kernel:

- one Triton program instance handles a `(batch_index, channel_index)` pair
- recurrence over time remains explicit (`for t in range(seq_len)`)
- state dimension processed in blocks (`block_n`)
- output accumulation across state blocks

Representative snippet:

```python
pid = tl.program_id(0)
b = pid // d_inner
d = pid % d_inner

for n_start in tl.static_range(0, d_state, block_n):
    state = ...
    a_vals = ...
    for t in range(seq_len):
        u_t = ...
        delta_t = ...
        b_vals = ...
        c_vals = ...
        a_bar = tl.exp(delta_t * a_vals)
        state = a_bar * state + (delta_t * b_vals) * u_t
        contrib = tl.sum(c_vals * state, axis=0)
```

This is mathematically the same recurrence as the reference path; the speedup comes from fused kernel execution and lower Python/dispatcher overhead.

### Line-by-line walkthrough landmarks

- `_selective_scan_fwd_kernel`: map Triton program ID to `(b, d)`.
- `_selective_scan_fwd_kernel`: load channel-local `A` slice and initialize state.
- `_selective_scan_fwd_kernel`: run the per-token recurrence update.
- `_selective_scan_fwd_kernel`: accumulate output contributions and add skip term.
- `_selective_scan_fwd_kernel`: optionally store final recurrent state.

Code section: [`mamba4080/ops/selective_scan_triton.py`](mamba4080/ops/selective_scan_triton.py)

## 9.3 Backward recurrence anatomy

Backward replays the forward state trajectory in float32 for stability, then scans backward over time.

Core pattern:

```python
for t in range(seq_len - 1, -1, -1):
    grad_state = grad_state_next + gy_t.unsqueeze(-1) * C_t
    grad_C[:, t] = gy_t.unsqueeze(-1) * x_t
    grad_u[:, t] += sum(grad_state * (delta_t * B_t), dim=-1)
    grad_B[:, t] = grad_state * (delta_t * u_t)
    grad_delta[:, t] = sum(grad_state * (...), dim=-1)
    grad_A += sum(grad_state * (...), dim=0)
    grad_state_next = grad_state * A_bar_t
```

Important points:

- gradient math is explicit, not delegated to opaque black-box kernels
- compute promoted to float32 in backward for stability
- returned grads cast back to original dtypes

Code path to inspect: `_SelectiveScanTritonFunction.backward` in
[`mamba4080/ops/selective_scan_triton.py`](mamba4080/ops/selective_scan_triton.py)

## 9.4 Safety and fallback behavior

Before launching Triton, code checks:

1. Triton installed
2. compiler availability (`gcc` or `clang`)
3. CUDA device
4. supported dtype (`fp16`/`fp32`)
5. device consistency

Fallback records a reason and routes to vectorized implementation.

```python
if u.dtype not in (torch.float16, torch.float32):
    _warn_fallback(f"unsupported dtype {u.dtype}")
    return selective_scan_vectorized(...)
```

Code: [`mamba4080/ops/selective_scan_triton.py`](mamba4080/ops/selective_scan_triton.py)

<div class="box">
  <div class="box-title">Why this design is practical</div>
  You get acceleration where possible and guaranteed correctness everywhere. The fallback path keeps execution reliable when environment or dtype constraints are not met.
</div>

<details markdown="1">
<summary>Optional deep dive: Why not make the entire backward Triton-native immediately?</summary>

The current explicit backward makes gradient logic auditable and easy to parity-test against vectorized reference. For a correctness-first project, this is a sensible tradeoff before deeper kernel fusion.

</details>

# 11. Known limitation: bf16 Triton scan fallback

Current behavior in this codebase:

- Triton selective scan path accepts `fp16` and `fp32` inputs.
- If scan inputs are bf16, dispatcher falls back to vectorized PyTorch scan.

Reason in code:

```python
if u.dtype not in (torch.float16, torch.float32):
    _warn_fallback(f"unsupported dtype {u.dtype}")
    return selective_scan_vectorized(...)
```

Code: [`mamba4080/ops/selective_scan_triton.py`](mamba4080/ops/selective_scan_triton.py)

Practical interpretation:

- `triton + fp32`: real kernel speedup path
- `triton + bf16`: correct behavior, but scan kernel itself is not accelerated

This is why benchmark reports include both `triton_active` and `fallback_reason` fields.

# 18. Appendix B: optional deep-dive bridges

<details markdown="1">
<summary>Bridge: Discretization nuance and \(\Delta_t\)</summary>

In practice, \(\Delta_t\) behaves like an adaptive time step per token/channel. Larger values can speed up decay in some modes; smaller values retain memory longer. This is one way to read Mamba's selectivity as dynamic memory timescale control.

</details>

<details markdown="1">
<summary>Bridge: Why both conv and recurrent memory?</summary>

Depthwise causal convolution gives a short local receptive field with parallel token mixing. Recurrent SSM gives long memory with linear-time state updates. Their combination is one of Mamba's strengths.

</details>

<details markdown="1">
<summary>Bridge: Why include finite-difference tests?</summary>

Parity between two implementations can still be jointly wrong. Finite-difference checks provide an independent sanity check for local gradients of core parameters.

Code: [`mamba4080/tests/test_selective_scan_math.py`](mamba4080/tests/test_selective_scan_math.py)

</details>

# 19. Conclusion

Mamba is easiest to internalize if you keep three layers in your head simultaneously:

1. **Idea layer**: dynamic, content-dependent memory in linear-time recurrence.
2. **Math layer**: selective state update with token-conditioned \(\Delta_t, B_t, C_t\).
3. **Systems layer**: reference path for correctness, vectorized path for baseline speed, Triton path for GPU acceleration with safe fallback.

This project is structured around exactly those layers. If you follow the equations and snippets above, you can move from theoretical understanding to implementation-level confidence, including where and why RTX 4080-specific tuning matters.
